# Financial Lobster · AI 项目开发经验沉淀

> 一个飞书机器人：用户发文件 → 下载解析 → LLM 分析出财务摘要报告 → 支持多轮追问。
> 从 0 到现在（33 次提交，6.06–6.11）踩过的坑与沉淀下来的工程经验，聚焦「做 AI 应用」这件事。

---

## 一、LLM 长文档处理：Map-Reduce 是核心骨架

单次调用塞不下整个文档，这是 AI 应用最先撞到的墙。本项目的解法是经典的 **Map-Reduce**，但落地时每一步都有坑：

- **Map（分片分析）**：把文档按「逻辑页」打包成片段，每片段独立调 LLM 抽取客观信息。
  - 关键决策：**页数才是绑定约束，不是字符数**。`build_chunks` 用 `max_pages` 封顶，超出的页不静默丢弃，而是标记 `truncated`，在报告里显式告诉用户「只分析了前 N 页」。*静默截断是 AI 应用最隐蔽的 bug——用户以为分析了全文，其实没有。*
  - Map 阶段并发（`asyncio.gather` + `Semaphore`），但并发数要保守（默认 4），因为 LLM 平台几乎都有速率限制。
- **Reduce（归并合成）**：这里最容易翻车。**不能把所有片段笔记一次性塞进单次合成调用**——片段一多就会超模型输入上限。
  - 解法是 **分层归并（hierarchical merge）**：笔记超过 `group_size` 或拼接后超过字符预算时，先分组合并、逐层收敛，最后才终合（`reduce_notes`）。
  - 防死循环：如果分组后组数不比原来少（单段已接近预算上限），就停止分层，交给最终合成尽力而为。

**经验**：Map-Reduce 不是写两个函数就完了，真正的工程量在「每一层的输入都不能超模型上限」这个不变量上。

## 二、上下文窗口 = 字符预算管理，要处处设防

模型的输入/输出长度上限是物理约束，整个系统要围绕它做「预算分配」：

- **输入侧**：`_pack_groups` 按「条数上限 + 字符预算」双约束打包；最终合成前还预留 2000 字符给报告模板和文件元信息（`fit_budget = max_input_chars - 2000`），避免「笔记拼好正好卡上限，再加模板就爆」。
- **输出侧**：模型输出被 `max_tokens` 截断时（`finish_reason == "length"`），用 `complete_until_done` **自动续写**——把已生成内容作为 assistant 轮回填，要求模型「从中断处接着写、不重复」，逐轮拼接直到 `finish_reason == "stop"`。
  - 隐藏代价：续写时回填内容会逐轮增大输入，所以 `max_rounds` 同时是「防跑飞」和「防输入超限」的硬上限。
- **检索侧**：拼接检索片段时按 `max_chars` 边累加边截断（`retrieve_by_embedding` / `retrieve_pages`）。

**经验**：长度上限不是单点检查，而是贯穿 map / reduce / 续写 / 检索的全局预算系统。任何一处忘了设防，就会在某个大文件上炸。

## 三、RAG / 追问：双路检索 + 跨语言是真问题

追问功能（基于已上传文件问答）用的是检索增强（RAG），踩到的关键坑：

- **中英混排材料下，关键词检索会失效**：中文问题 vs 英文正文字面零重合，TF 打分直接挂掉。这是实打实的业务痛点，不是理论问题。
  - 解法：**多语言 embedding 做语义检索**，关键词检索（`retrieve_pages`，基于 TF 计数）作为兜底。两路并存，向量失败/无缓存就回退关键词。
- **切块策略**：检索粒度要比「整页」更细（`qa_embedding_chunk_chars=700`），且相邻块要 **overlap**（120 字符），避免答案恰好被切在块边界而漏召回。
- **向量空间一致性**：查询必须用「该文件入库时所用的同一个 embedding 模型」，否则向量空间错配、相似度全是噪声。所以缓存里记了 `model` 字段，查询时强制读出来用。
- **两级文件选择**：先按问题与各文件关键字的重合度选最相关文件，都不命中就用最近活跃文件。

**经验**：RAG 不是「embedding 一把梭」。检索失败要有兜底路径；embedding 的模型、维度、切块、overlap 每一个都会实打实影响召回质量。

## 四、LLM 调用的鲁棒性：把「会失败」当默认假设

调外部 LLM API = 调一个会超时、会限流、会欠费、会返回半截的远程服务。本项目的防御层：

1. **传输层重试**：`httpx.TransportError`（连接超时/读超时/连接错误等瞬时故障）才重试，已分类的业务错误不重试（避免把配置错误反复撞）。
2. **错误分类 + 友好提示**：`_classify_llm_error` 不依赖具体平台的 JSON 结构，而是按「跨平台通用关键词」（arrearage / insufficient_quota / rate limit …）把错误归类成 billing/auth/rate_limit/model/server，再给用户**可执行的中文提示**而非生吞 500。
   - 细节坑：阿里百炼免费额度耗尽返回 **403**，极易被误判成鉴权失败——所以 billing 判定必须排在 auth 之前，否则触发不了 fallback。
   - `httpx` 默认的 `raise_for_status()` 会丢弃响应体，导致欠费/鉴权信息丢失，自己实现 `_raise_for_status_with_body` 先存 body 再分类。
3. **模型 Fallback**：`FallbackProvider` 在主模型额度耗尽（category=billing）时**永久切到下一个备用模型**，游标只前进不回退（已知耗尽的不再重撞，并发场景加锁保护推进）。其余错误不切换，直接上抛。
4. **输出截断自动续写**（见第二节）。

**经验**：AI 应用的「正确性」一半在 prompt，另一半在「API 失败时用户看到什么」。错误分类 + 友好降级提示是体验的分水岭。

## 五、缓存设计：省 token 的核心，key 怎么设计是门道

LLM 调用按 token 收费且慢，缓存直接决定成本和速度。两层缓存：

- **分片分析缓存**（`AnalysisCache`）：缓存每个片段的分析结果。
  - **缓存 key 故意只描述「输入内容」（prompt_version + chunk_chars + file_hash + chunk_index + chunk_hash），不含 model/temperature/max_tokens**。这样换模型仍能命中——因为「客观信息抽取」任务不同模型结果近似可互换。要主动失效就 bump `prompt_version`。这是一个有意识的取舍。
  - 文件按 `sha256` 内容哈希做 key，相同内容重复上传直接复用。
- **向量缓存**（按 file_hash 命名的 JSON）：同文件重传不重算 embedding，并记录入库所用模型。

**经验**：缓存 key 的设计是「你认为什么变了结果就该变」的显式声明。把模型踢出 key 是为了跨模型复用，这种取舍要写清楚（代码里专门留了注释），否则后人不敢动。

## 六、Prompt 工程：约束比能力更重要（尤其金融/合规场景）

面向投顾的财务材料分析，**幻觉是致命的**。System Prompt 的设计全在「划边界」：

- 反复强调：只反馈材料中出现的客观内容，**不臆造、不补全、不推断**。
- 所有数据/观点**强制标注来源页码**（`[第 X 页]`），让结论可溯源——这也是后续追问能引用页码的基础。
- 没依据就写「未在材料中发现」「不足以判断」，而不是硬编。
- **明确禁止输出投资建议/审计意见/法律结论**（合规红线）。
- 报告结构用「必须包含且只包含以下模块」固化输出格式，客观数据强制走 Markdown 表格。

**经验**：在专业/合规领域，prompt 的主要工作不是「让模型更聪明」，而是「框住模型不要乱说」。来源标注（page citation）既是合规要求，也是 RAG 追问的地基。

## 七、系统工程：AI 只是其中一环

一个能用的 AI 应用，AI 调用之外的工程量同样大：

- **事件幂等**：飞书没收到 ACK 会重推同一事件。用 **文件系统 `O_EXCL` 原子创建** 实现去重认领（OS 层原子，天然适配多线程/多进程，无需额外锁），放在处理链最前面，连重复 token 消耗都省掉。
- **尽力而为 vs 必须成功**：进度通知发送失败**不能中断**已在跑的分析（`notify` 吞异常）；管理员旁路推送失败也只记日志。区分「主流程」和「锦上添花」，后者永远不能拖垮前者。
- **渐进式可用**：报告一发出就**立即以空 embedding 登记文件**，用户能马上追问（先走关键词检索），后台 embedding 算完再补写路径无缝升级为向量检索。不让用户等向量算完。
- **多格式解析统一抽象**：PDF/Word/CSV/Excel 都归一成 `ParsedDocument` + 逻辑页。非分页格式（Word/表格）没有物理页概念，就按字符预算切「逻辑页」，让追问仍能给出有意义的页码引用。CSV 编码兜底（utf-8-sig → gbk → latin-1）。
- **任务状态机 + 持久化**：每一步（downloading/parsing/analyzing/reporting/succeeded/failed）都落 `TaskStore`，便于排查「卡在哪一步」。
- **全链路可观测**：追问把检索方式、命中页、历史轮数、最终 prompt 全文都打日志，效果不好时能复盘。

**经验**：用户感知的「这个 AI 好不好用」，70% 取决于 AI 之外——幂等、降级、渐进可用、状态可查。

## 八、配置驱动：让运维不改代码也能调

几乎所有「数字」都进了 `Settings`（pydantic-settings + .env）：分片大小、最大页数、并发数、归并组大小、字符预算、Top-K、overlap、超时、各种目录……

- **chat 与 embedding 可走不同平台/账号**：embedding 留空则复用 chat 配置，填写则分流到独立（如免费）账号。这是为了「chat 额度耗尽时把向量算力分流」的真实运维需求。
- 备用模型链、prompt 版本号都可配。

**经验**：AI 应用的「正确参数」是调出来的，不是想出来的。把 chunk 大小、并发、Top-K 这类需要反复试的旋钮做成配置，能极大加快迭代——改 .env 重启，而不是改代码重发。

---

## 九、一句话总结

> **做 LLM 应用，模型调用本身可能只占 20% 的工作量。**
> 剩下 80% 是：长度预算管理、Map-Reduce 收敛、检索召回质量、失败分类与降级、缓存省钱、prompt 划边界、幂等与渐进可用。
> 把「模型一定会超时/限流/欠费/截断/幻觉」当默认假设来设计，应用才真正可用。
