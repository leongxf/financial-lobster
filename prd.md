# 财务文件智能分析多入口产品 PRD

## 1. 背景与目标

四大等专业服务机构的投资建议、交易咨询和经营分析人员在项目中需要快速阅读大量财务或经营相关材料。当前常见痛点包括：文件格式多、材料篇幅长、观点分散、数据位置难找、人工整理耗时、投资判断前缺少统一的客观材料摘要。

本产品面向四大等专业服务机构的投资建议、交易咨询和经营分析人员，提供一个在企业网络安全管控场景下仍可使用的外部网络个人 AI 助理。MVP 支持通过本机常驻飞书长连接 worker 接收财务或经营相关资源文件，由本机进程完成文件存储、解析、结构化抽取和大模型分析，最终通过飞书 Markdown 消息返回可读的财务摘要、客观数据表和全文概要。

当前用户反馈显示，部分使用环境禁止微信传输文件，但允许通过飞书上传。因此 MVP 入口建议从“微信小程序优先”调整为“飞书上传优先”，同时在架构上抽象入口适配层，避免核心分析能力被某一个平台绑定。

MVP 第一阶段不做银行流水、发票、合同或科目余额表等专项财务分析，优先实现第一个 AI 助理能力：“用户上传财务或经营相关资源文件后生成财务摘要”。摘要应帮助用户从投资分析和风险判定视角快速整理材料中的客观信息，包括文件概览、一句话摘要、财务要点摘要、全文概要、客观数据表、观点位置和可复核引用。

如果第一个材料分析能力验证可用，产品后续会扩展更多面向投资建议、交易咨询和经营分析岗位的个人 AI 助理能力。因此系统必须从一开始保留能力扩展口子，避免把架构写死为单一文件摘要工具。

### 产品目标

- 降低投资建议和交易咨询人员整理财务/经营材料的门槛，优先适配用户当前能使用的上传入口。
- 自动识别文件内容，优先输出财务摘要、全文观点概要、客观数据提取和原文引用。
- 形成可追溯的分析任务记录，支持后续复核、下载和共享。
- 将上传入口、用户身份、消息通知与核心分析能力解耦，为后续接入微信、Web、企业微信、钉钉、邮箱、网盘等入口保留扩展空间。
- 将“材料分析”抽象为一个可插拔 AI 能力模块，为后续扩展更多个人助理能力保留统一入口、统一任务、统一权限、统一配置和统一结果展示框架。

### 非目标

- 第一阶段不替代专业财务软件的账务处理、凭证生成和报税申报。
- 第一阶段不直接连接企业网银、税务局、ERP、金蝶、用友等系统。
- 第一阶段不承诺大模型结论作为法律、审计或税务最终意见，所有结论需人工复核。
- 第一阶段不做复杂自主 Agent，不让模型自行决定业务流程；优先用成熟框架编排确定性流程。

## 2. 可行性判断与入口策略

该方案可行，但入口不应固定为微信小程序。对于个人使用场景，最重要的是选择用户当前网络和设备环境允许的上传路径。如果用户实际发现飞书可以上传文件，MVP 应优先选择飞书作为第一个上传入口。

推荐采用“入口适配层 + 后端任务系统 + 文件解析服务 + 大模型分析服务 + 结果持久化”的架构。飞书、微信小程序、Web、企业微信等入口只负责身份识别、文件接入、消息触达和结果展示，核心文件解析和 AI 分析流程保持不变。

### 可行的原因

- 飞书支持文件消息、机器人、应用、云文档等能力，适合作为个人或团队环境中的文件上传入口。
- 微信小程序支持选择和上传聊天文件、本地文件、图片等，适合微信环境未被限制的用户。
- 财务文件分析通常是异步任务，适合通过后端排队处理，再由入口端轮询或消息通知提醒用户查看结果。
- 大模型对财务文件摘要、分类、异常解释、自然语言报告生成有较好适配度。
- OCR、Excel/CSV/PDF 解析、MySQL、Redis、任务队列等基础能力已经成熟。

### 关键限制

- 不同入口的文件能力、登录机制、消息通知和审核规则不同，必须通过入口适配层隔离差异。
- 微信小程序有文件大小、上传链路、运行环境和接口域名限制；飞书也有文件 API 权限、机器人权限、文件下载有效期和应用审核限制。
- 如果用户环境禁止某类工具传输文件，不应设计绕过安全管控的能力，而应允许用户切换到被允许的入口。
- 用户企业内存在安全机制，会拦截 H5/Web 上传文件链路，因此 MVP 不采用 H5/Web 作为上传入口。MVP 结果也优先通过飞书 Markdown 消息返回，H5 结果页作为后续增强。
- 财务数据敏感，必须重点处理数据安全、权限、加密、日志脱敏、供应商合规和数据留存策略。
- 大模型结果存在幻觉风险，产品必须展示数据来源、置信度、可复核明细和免责声明。
- 财务摘要必须严格遵循“只反馈材料中的客观结果，不臆造任何材料内容”的原则；凡无法从原文确认的信息，必须标记为“未在材料中发现”或“不足以判断”。
- 对复杂扫描件、手写票据、低质量 PDF 的识别质量取决于 OCR 和版面解析能力。

### MVP 入口结论

- 第一优先级：飞书机器人消息上传。原因是用户当前环境已验证飞书可用，且 H5/Web 上传会被企业安全机制拦截。
- 第二优先级：飞书应用能力。仅在机器人消息无法满足文件接收、权限或通知需求时评估。
- 第三优先级：微信小程序。适合微信未被限制的用户，但不作为 MVP 入口。
- H5/Web 上传：不作为 MVP 上传入口。H5 结果页不作为 MVP 必做能力，后续多人稳定使用时再评估。
- 架构原则：所有入口统一转成标准的 `UploadRequest` 和 `AnalysisTask`，后续核心层不感知文件来自飞书、微信还是 Web。

## 3. 用户与场景

### 目标用户

- 四大投资建议人员：希望快速整理材料中的财务和经营数据，辅助判断是否值得投资。
- 四大交易咨询人员：希望快速定位材料中的关键观点、章节和数据来源，减少人工阅读和摘录时间。
- 经营分析人员：希望从材料中提取客观经营指标，用于后续分析和风险判断。
- 项目经理或复核人员：希望快速检查摘要是否有原文依据，避免模型臆造影响判断。

### 典型文件

- Excel/CSV：费用明细、银行流水、销售台账、库存台账、科目余额表。
- PDF：合同、发票、对账单、银行回单、审计报告。
- 图片：发票照片、收据、付款截图、报销凭证。
- Word：合同、说明文件、财务制度。

## 4. 用户故事

### 文件上传

作为投资建议人员，我希望通过当前可用的入口上传 Excel、CSV 或可解析 PDF 文件，以便系统自动开始整理材料。

验收标准：

- MVP 支持通过飞书上传单个文件。
- 后续可扩展微信小程序、Web、企业微信、钉钉、邮箱等入口。
- 系统展示支持的文件类型、大小限制和预计分析时间。
- 上传失败时展示明确原因，例如文件过大、格式不支持、网络失败。
- 上传成功后生成一个分析任务，并进入处理中状态。

### 分析任务管理

作为投资建议人员，我希望查看历史分析任务，以便追踪每个文件的处理状态和分析结果。

验收标准：

- 用户可以查看任务列表，包括文件名、上传时间、状态和结果摘要。
- 用户可以看到任务来源入口，例如飞书、微信小程序或 Web。
- 任务状态至少包括：待上传、已上传、解析中、分析中、成功、失败。
- 失败任务展示失败原因，并支持重新提交。

### 文件内容解析

作为投资建议人员，我希望系统能识别文件中的关键字段和表格内容，以便减少人工整理。

验收标准：

- Excel/CSV 文件可以提取工作表、表头、行列数据和基础统计信息。
- 可解析 PDF 可以提取内嵌文本内容。
- 图片、扫描件和图片型 PDF 后续通过 OCR 支持，不作为 MVP 第一阶段能力。
- 系统保留原始文件、解析文本和结构化结果之间的关联。
- 解析质量较低时，系统提示用户结果需要重点复核。

### 智能分析报告

作为四大投资建议人员，我希望系统基于文件内容生成可读的财务摘要，以便客观理解材料中与投资分析和风险判定相关的信息。

验收标准：

- 报告包含文件概览、一句话摘要、财务要点摘要、全文概要和客观数据表。
- 全文概要需要说明全文有哪些主要观点，以及每个观点出现在全文的哪些章节、页码、Sheet 或段落。
- 报告需要提取材料中的财务或经营相关数据，用作后续投资分析和风险判定。
- 客观数据表需要尽量结构化展示指标名、数值、单位、期间、主体和来源位置。
- 报告中的摘要、观点和数据必须引用原始数据位置或明细。
- 系统不得臆造材料中不存在的信息；无法确认的信息必须明确标记。
- 系统明确标识“AI 生成，需人工复核”。
- 用户可以复制或下载报告。

### 后续专项风险分析

作为投资建议人员，我希望系统在后续版本中基于客观数据指出潜在风险线索，以便辅助判断是否投资。

验收标准：

- 系统能识别金额异常、重复记录、日期异常、供应商异常、税率异常等常见问题。
- 每个风险点展示风险原因、相关数据、严重程度和建议动作。
- 用户可以将风险点标记为已确认、误报或待跟进。

说明：该能力不作为 MVP 第一阶段核心目标。第一阶段只整理材料中的客观信息、全文观点和引用依据，专项异常检测和财务风险规则放到后续迭代。

### 结果复核

作为投资建议人员，我希望查看 AI 摘要对应的原始数据，以便判断结论是否可信。

验收标准：

- 报告中的关键结论可以关联到原始文本、表格行或文件页码。
- 用户可以查看解析后的结构化数据。
- 用户可以反馈“准确”“不准确”或补充说明。

### 安全与权限

作为企业管理者，我希望财务文件只有授权人员可以访问，以便保护敏感数据。

验收标准：

- 用户必须通过入口账号或系统账号完成身份识别后才能上传和查看文件。
- 用户只能查看自己创建的任务。
- 企业版后续支持组织、角色和成员权限。
- 文件下载链接必须有有效期和访问权限校验。

### 入口切换

作为个人用户，我希望在微信或 H5/Web 上传不可用时可以通过飞书机器人上传文件，并直接在飞书中收到 Markdown 格式分析结果。

验收标准：

- 系统支持飞书机器人作为 MVP 主入口，并保留后续可扩展入口。
- 同一个用户在不同入口产生的任务可以在统一账户下归集，或在未打通账号时明确展示为不同来源。
- 新增入口时不需要改动文件解析、大模型分析、报告生成等核心逻辑。

## 5. MVP 范围

### 第一阶段必须支持

- 飞书机器人消息上传。
- 单文件上传。
- 支持 Excel、CSV、可解析 PDF 文件。
- 文件存储到 ECS 本地临时目录，按留存配置自动清理。
- 后端异步创建分析任务。
- 文件解析：Excel/CSV 结构化解析，可解析 PDF 文本提取。
- 大模型分析：文件概览、一句话摘要、财务要点摘要、全文概要、客观数据表、观点位置、客观数据提取。
- 分析任务列表和详情页。
- 分析失败重试。
- 基础日志、错误监控和操作审计。
- 入口适配层抽象，至少支持飞书机器人入口，预留微信小程序、企业微信、钉钉等扩展点。

### 第一阶段暂不支持

- 多文件合并分析。
- 企业组织与多人协作。
- 多入口账号自动合并。
- 对接 ERP、网银、税务和电子发票平台。
- 自动生成会计凭证。
- 复杂审批流。
- 私有化部署。

## 6. 核心流程

1. 用户在飞书中向机器人发送文件。
2. 飞书入口适配器接收事件，获取文件元信息和临时下载凭证。
3. 后端下载文件并保存到 ECS 本地临时目录。
4. 后端创建分析任务，记录文件元信息、用户标识和入口来源。
5. 入口适配器将外部平台用户映射为内部用户。
6. 后端任务队列触发解析服务。
7. 解析服务根据文件类型提取文本、表格和元数据。
8. 后端将解析结果切片、脱敏或压缩后提交给大模型。
9. 大模型返回结构化分析结果。
10. 后端保存摘要报告、客观数据提取结果、观点位置和引用关系。
11. 飞书入口适配器通过飞书消息通知用户查看结果。
12. 用户通过飞书 Markdown 消息查看财务摘要、全文概要、客观数据和原始数据引用。

## 7. 推荐架构

### 前端

- 飞书机器人入口：MVP 上传和结果返回入口，通过文件消息接收文件，通过 Markdown 文本返回处理状态和分析结果。
- H5 结果页：后续增强能力，用于多人稳定使用时展示更复杂的报告。
- H5/Web 上传：不作为 MVP 上传入口。
- 微信小程序：作为后续可选入口，不再作为 MVP 唯一入口。
- 技术选择：MVP 推荐飞书机器人消息方案。后续如果需要微信端，再补微信小程序适配器。

### 入口适配层

- 负责对接不同平台的登录、文件事件、文件下载、消息通知和结果链接。
- 将外部入口统一转换为内部标准对象：`ExternalUser`、`UploadRequest`、`UploadedFile`、`AnalysisTask`。
- 每个入口独立实现适配器，例如 `FeishuAdapter`、`WechatMiniProgramAdapter`、`WebUploadAdapter`。
- 核心层只消费标准对象，不直接依赖飞书、微信或 Web 的 SDK。
- 新增入口时，只新增适配器和少量前端页面，不改文件解析、任务调度、模型分析和报告生成。

### AI 能力扩展层

系统需要预留从“材料分析”扩展到更多个人 AI 助理能力的口子。MVP 只实现 `material_financial_summary` 能力，但后端任务模型、Prompt 管理、结果存储和结果渲染不应只服务这一个能力。

核心设计：

- `Capability`：定义一个 AI 能力，例如材料财务摘要、会议纪要整理、尽调问题清单、资料核对、投委会材料草稿、数据表校验。
- `CapabilityVersion`：管理能力版本、Prompt 版本、输出 schema、启用状态和适用文件类型。
- `AssistantTask`：统一承载用户发起的 AI 任务，任务类型由 `capability_code` 区分。
- `ResultRenderer`：按能力类型渲染结果。MVP 先实现飞书 Markdown 模板，后续可扩展 H5 模板。
- `ProviderConfig`：统一管理模型供应商、模型参数和调用超时。
- `RetentionConfig`：统一管理不同任务类型的数据留存策略。

后续新增能力时，应优先新增能力配置、Prompt、输出 schema 和结果模板，而不是重写入口、任务队列、用户体系或模型调用层。

潜在扩展方向：

- 材料问答：围绕用户上传材料进行引用式问答。
- 尽调问题清单：基于材料客观缺口生成需要进一步询问的问题。
- 投资备忘录草稿：基于客观摘要生成可编辑初稿，但必须标注来源。
- 经营数据对比：对多份材料中的经营指标做横向对比。
- 风险线索整理：基于规则和引用生成风险线索，不输出投资建议。
- 工作清单生成：把材料中的待核实事项转成个人任务清单。

### 后端 API

- 负责用户、入口账号映射、文件元信息、任务状态、结果查询、权限校验。
- 推荐使用 Node.js、Python FastAPI、Java Spring Boot 中一种。若团队熟悉大模型和文件解析，Python FastAPI 更方便。

### 文件存储

- MVP 暂不接入阿里云 OSS，优先使用 ECS 本地临时目录保存原始文件和解析中间文件，让功能尽快跑通。
- 文件路径只在服务端保存，不暴露本地文件路径。
- 原始文件允许按日清理；具体留存天数由数据库配置控制。
- 后续当用户量、文件量或合规要求上升时，再迁移到阿里云 OSS。

### 异步任务

- 使用消息队列或任务队列处理文件解析和模型分析。
- 避免入口端等待长请求，提升稳定性。

### 文件解析

- CSV：使用 Python 标准库或 pandas 解析。
- Excel：使用 openpyxl/pandas 解析，保留 sheet、行列、表头和基础统计信息。
- PDF：使用 pypdf 提取内嵌文本，扫描件后续再走 OCR。
- 图片：MVP 暂不支持，后续使用 OCR 提取文字并保留识别置信度。

### 大模型分析

- 后端统一封装模型调用，不从飞书、微信、Web 等入口端直接调用模型。
- 将解析结果转成结构化提示词，要求模型输出 JSON 结构，后端做 schema 校验。
- MVP 大模型供应商暂定通义千问。
- 模型调用必须通过通用 `LLMProvider` 封装实现，不允许业务代码直接依赖某个供应商 SDK。
- 供应商差异通过配置解决，至少支持 `provider`、`baseURL`、`apiKey`、`appId`、`model`、`timeout`、`maxTokens`、`temperature`。
- 优先兼容 OpenAI-style Chat Completions 接口；若供应商接口不兼容，在 Provider 适配器内转换为统一请求和统一响应。
- AI 流程编排优先使用 LangGraph、LangChain 等成熟框架，不自研 Agent 框架。MVP 需要的是稳定的文件解析、摘要生成、结构化输出和校验流程，不需要复杂自主决策 Agent。
- 对财务或经营相关数据，优先用程序抽取，再让模型基于原文生成摘要、观点归纳和章节定位。
- 模型输出不得包含无引用依据的事实性判断。若模型无法在材料中找到依据，应输出“未在材料中发现”或“不足以判断”。

### LLM Provider 配置示例

```env
LLM_PROVIDER=qwen
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_API_KEY=replace-with-api-key
LLM_APP_ID=
LLM_MODEL=qwen-plus
LLM_TIMEOUT_MS=60000
LLM_MAX_TOKENS=4000
LLM_TEMPERATURE=0.2
```

业务层只调用统一接口：

```text
LLMProvider.generateStructuredSummary(input) -> FinancialSummary
```

后续从通义千问切换到 DeepSeek、火山方舟、腾讯混元或 OpenAI 时，应优先只修改环境变量；只有接口协议不兼容时，才新增或调整 Provider 适配器。

### 数据库

- 使用 MySQL 保存用户、文件、任务、解析结果、摘要报告、客观数据提取结果、观点位置、能力配置、系统配置和反馈记录。
- 优先使用支持 JSON 字段的 MySQL 版本，方便保存结构化报告、模型输出和配置。
- 敏感字段加密存储，日志脱敏。

### 安全

- 入口账号与内部用户映射，例如飞书 open_id、union_id、tenant_key，微信 openid/unionid，Web 账号。
- HTTPS、域名备案、接口白名单。
- 本地文件目录禁止公开访问，只允许后端服务读取。
- 服务端权限校验。
- 文件病毒扫描或基础安全检测。
- 敏感日志脱敏。
- 数据保留和删除策略。

## 8. 服务采购清单

### 容量假设

MVP 目标用户规模在 100 人以内，属于低并发、轻量使用场景。系统设计应优先保证产品闭环、数据安全和任务可恢复，不需要一开始采购复杂的托管消息队列、数据库集群、Worker 集群或 CDN。

建议按以下规模估算：

- 注册用户：<= 100 人。
- 日活用户：<= 30 人。
- 单日上传文件：<= 200 个。
- 同时分析任务：<= 5 个。
- 单文件大小：MVP 建议限制在 20MB 以内。

### 必买或必备

- 飞书开放平台应用：用于接收文件消息、识别用户、发送分析结果通知。
- 服务域名：用于接收飞书事件回调和提供后端 API，不作为用户上传入口。
- 已备案域名与 HTTPS 证书：中国大陆生产环境必须准备。
- 云服务器：MVP 可先购买一台阿里云 ECS，用 Docker Compose 承载后端 API、任务 Worker、数据库、Redis 和 Nginx。
- 本地临时文件存储：MVP 文件先存 ECS 本地磁盘，允许按日自动清理，先保证功能跑通。
- 数据库：使用 Docker 内的 MySQL，保存用户、任务、结果、配置和审计记录。
- Redis：用于缓存、Celery 任务队列和任务锁。
- 大模型 API：MVP 暂定通义千问，用于财务摘要、全文概要、观点定位和客观数据整理；实现上必须使用通用 Provider 封装，后续可通过配置切换 OpenAI、DeepSeek、腾讯混元、火山方舟、智谱等。
- 日志与监控：MVP 使用容器标准输出、文件日志、结构化 JSON 日志和飞书错误告警，记录错误、接口耗时、任务失败率和模型调用成本，暂不接入复杂日志平台。

### 建议购买

- OCR 服务：按需购买，用于识别扫描件、图片和图片型 PDF，例如腾讯云 OCR、阿里云 OCR、百度 OCR。若 MVP 只支持 Excel、CSV、可解析 PDF，可暂不购买。
- 阿里云 OSS：MVP 暂不使用。等文件量、留存要求或备份要求上来后再迁移。
- 托管消息队列：例如腾讯云 CMQ/TDMQ、阿里云 MNS/RabbitMQ。MVP 使用 Celery + Redis，不购买托管消息队列。
- CDN：如果后续需要加速报告或静态资源访问。
- 内容安全或文件安全扫描：降低恶意文件和敏感内容风险。
- 短信或邮件服务：企业版可用于通知，但 MVP 阶段可优先使用飞书消息通知。
- 微信小程序账号：如果后续要恢复微信端入口，再购买或启用。

### MVP 运行建议

第一阶段可以采用本机常驻进程的最低成本方案：

- `feishu_ws_worker`：飞书长连接 worker，接收文件消息、下载文件、解析材料、调用模型并回复 Markdown 报告。
- `local-storage`：本机临时文件目录，用于保存上传文件和解析中间文件，按配置自动清理。
- `mysql`：可选。MVP Spike 阶段可以先不启用数据库，后续需要历史任务、配置和留存策略时再启用。
- `redis/celery`：可选。MVP Spike 阶段可以先用进程内后台线程，后续任务变慢或用户变多时再启用。

本机方案适合验证第一个材料分析能力，不需要阿里云、Nginx、公网域名或 H5 页面。限制是必须保持本机在线，进程停止后无法接收飞书事件，不适合多人稳定使用。

后续多人稳定使用时，再迁移到阿里云中国大陆 ECS + Docker Compose。建议初始配置：4 核 8GB、100GB 磁盘、5Mbps 固定带宽，并配置自动快照和数据库定时备份。

### 供应商组合建议

后续多人稳定使用阶段，优先选择阿里云中国大陆区域：

- 腾讯云组合：飞书机器人 + 腾讯云 COS + CVM/云函数 + 可选 OCR + 混元/DeepSeek 接入 + MySQL。
- 阿里云组合：飞书机器人 + ECS + Docker Compose + MySQL + Redis/Celery + 本地临时文件存储 + 可选 OCR + 通义千问。

MVP Spike 暂定本机常驻进程。后续如需要多人稳定使用，再迁移到阿里云中国大陆区域 + ECS + Docker Compose。

## 10. 交互设计

### 飞书机器人入口

- 用户向飞书机器人发送 Excel、CSV 或可解析 PDF 文件。
- 系统立即回复首轮确认，包含**使用的模型名称**（如 `qwen-plus`）和当前处理步骤。
- 处理过程中按步骤推送进度：下载完成、文本提取完成、每个分析片段、合成报告等。
- 系统在分析完成后通过飞书 Markdown 消息发送分析结果。
- 若报告内容过长，系统拆分多条消息发送。

### 飞书结果消息

- 展示文件名称、文件类型、分析完成时间。
- 展示一句话摘要。
- 展示 Markdown 格式摘要。
- 支持用户通过关键词或按钮反馈结果是否有用。

### 非 MVP 页面

- MVP 阶段不提供 H5/Web 上传页。
- MVP 阶段不提供 H5 报告详情页。
- 后续若用户环境允许，再评估 H5 报告页、Web 管理台和任务列表页。

### 后续管理台页面

- 任务列表页：展示历史任务、按状态筛选、失败任务重试。
- 任务详情页：展示文件信息、处理状态、失败原因和重试按钮。
- 报告详情页：展示文件概览、一句话摘要、财务要点摘要、全文概要、客观数据表、原始数据引用、复制、下载和反馈入口。
- 以上页面不属于 MVP，只有在确认用户环境允许访问 Web 后再开发。

## 12. 关键风控与合规

- 明确告知用户文件会被上传到云端并用于 AI 分析。
- 用户上传前需同意隐私政策和用户协议。
- 允许将文件内容发送给第三方大模型 API 进行分析，但必须在隐私政策、用户协议和上传提示中明确告知。
- 默认不把用户文件用于模型训练，除非用户明确授权，并优先选择承诺 API 数据不用于训练的模型供应商。
- 大模型供应商必须确认数据处理、留存和跨境策略。
- 对财务文件、身份证号、银行卡号、手机号等敏感信息进行脱敏展示或限制日志记录。
- 提供删除文件和删除分析结果能力。
- 数据留存策略通过数据库配置控制，默认原始文件保留 30 天、解析文本保留 30 天、分析报告保留 180 天。
- 服务内执行定时清理任务，根据数据库配置自动清理过期原始文件、解析文本和分析报告。
- 自动清理任务必须记录清理日志，包含配置快照、清理数量、执行时间、状态和失败原因。
- 企业版应支持操作审计、访问控制和数据导出。

## 13. MVP 验收指标

- 飞书入口单文件上传成功率 >= 98%。
- 常见 Excel/CSV 解析成功率 >= 95%。
- 可解析 PDF 文本提取成功率 >= 90%，低质量或扫描件 PDF 除外。
- 10MB 以内文件从上传完成到生成报告的 P95 时间 <= 120 秒。
- 分析任务失败时必须有可读错误原因。
- 用户可以在飞书中完成上传，并在分析完成后收到结果通知。
- 报告中的重要摘要和关键信息尽量包含数据来源引用。
